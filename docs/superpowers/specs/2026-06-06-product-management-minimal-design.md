# 商品管理最小版设计

## 背景

当前前台工作台已经有 `对话`、`素材库`、`项目管理`、`个人信息` 等侧栏入口，但缺少面向普通用户的商品管理界面。后端商品模块的基础 CRUD 接口已经存在，前端尚未提供可直接使用的商品列表与维护入口。

本次目标是在前台左侧栏新增一个 `商品` 入口，进入后可查看当前用户拥有的商品，并支持最小版的新增、编辑、删除流程。

## 目标

- 在前台左侧栏新增 `商品` 入口
- 点击后进入“我的商品”视图
- 展示当前登录用户拥有的商品
- 支持新增商品
- 支持编辑商品
- 支持删除商品
- 新增/编辑统一使用一套表单弹窗

## 非目标

本次明确不做以下内容：

- 商品素材上传与商品素材绑定
- 商品自动解析、手动解析、解析进度展示
- 分类树选择与多级分类维护
- 平台商品抓取
- 卖点、规格、标签等复杂结构化字段编辑
- 独立前台路由页或独立商品子应用
- 与对话生成链路的商品联动

## 字段范围

第一版商品表单只包含以下字段：

- `name`：商品名称，必填
- `brand`：品牌，选填
- `price`：价格，选填
- `main_image_url`：主图链接，选填
- `description`：商品描述，选填

说明：

- `price` 在前端表单中按字符串管理，提交时转换为数字或 `null`
- `main_image_url` 做基础 URL 格式校验，但不做图片可访问性校验
- 复杂字段如 `selling_points`、`specs`、`tags` 保留为空，不进入本次 UI

## 用户流程

### 1. 查看商品

用户点击左侧栏的 `商品` 后，主区域切换到“我的商品”页面，页面加载当前用户商品列表。

页面优先显示：

- 商品主图
- 商品名称
- 品牌
- 价格
- 简短描述
- 创建时间

每个商品卡片提供：

- `编辑`
- `删除`

### 2. 新增商品

用户点击页面右上角 `新增商品` 按钮后，打开表单弹窗。

用户填写最小字段后点击保存：

- 表单校验通过
- 调用创建接口
- 成功后关闭弹窗
- 刷新商品列表

### 3. 编辑商品

用户点击商品卡片上的 `编辑` 后，打开表单弹窗并预填当前商品数据。

保存成功后：

- 关闭弹窗
- 刷新商品列表

### 4. 删除商品

用户点击商品卡片上的 `删除` 后，先出现确认提示。

确认后：

- 调用删除接口
- 成功后刷新列表

## 页面结构

### 侧栏

在现有前台侧栏中新增一个 `商品` 按钮，风格与 `素材库`、`项目管理` 保持一致，仍然使用 `activeView` 切换机制，不新增独立前台路由。

### 商品页

页面分为四部分：

1. 顶部标题区
- 标题：`我的商品`
- 副标题：说明这里只管理当前账号的商品
- 右侧操作：`新增商品`

2. 列表区
- 第一版使用卡片列表
- 兼容桌面与中小屏响应式布局

3. 空状态
- 当用户暂无商品时显示空状态文案
- 提供显眼的 `新增商品` 按钮

4. 表单弹窗
- 统一用于新增与编辑
- 标题根据场景显示 `新增商品` 或 `编辑商品`

## 前端架构设计

### 入口整合

复用现有前台工作台结构：

- 在 `Sidebar` 中新增 `商品` 入口
- 在 `MainLayout` 中为 `activeView === "products"` 增加渲染分支
- 继续复用 `useAppStore` 中的 `activeView`

### 组件拆分

建议新增两个组件：

- `ProductManager`
  - 负责商品列表加载、空状态、弹窗开关、删除交互
- `ProductFormModal`
  - 负责新增/编辑表单的渲染、校验、提交

这样可以把页面状态管理和表单逻辑拆开，避免单文件过大。

### 服务层

新增 `frontend/src/services/product.js`，封装：

- `listProducts(params)`
- `createProduct(data)`
- `updateProduct(productId, data)`
- `deleteProduct(productId)`

接口直接复用现有后端 `/v1/products`。

## 后端接口设计

本次默认复用已有接口：

- `GET /v1/products`
- `POST /v1/products`
- `PUT /v1/products/{product_id}`
- `DELETE /v1/products/{product_id}`

### 列表边界

需求是“当前用户拥有的商品”，但现有 `GET /v1/products` 在服务层描述中包含“当前用户商品 + 平台公共商品”。

第一版建议：

- 前端以 `is_public !== true` 作为过滤条件，只展示当前用户商品

若联调发现现有返回结构不足以稳定区分，可在下一轮补一个后端专用筛选参数，例如：

- `owned_only=true`

但本次设计默认不要求先改后端接口。

## 校验与错误处理

### 表单校验

- `name` 不能为空
- `price` 若填写则必须能转换为合法数字
- `main_image_url` 若填写则必须符合基础 URL 格式

### 请求失败处理

- 列表加载失败：页面内提示加载失败，并允许重试
- 新增/编辑失败：弹窗内或顶部提示失败信息，不自动关闭
- 删除失败：保留当前列表并弹出失败提示

## 视觉与交互原则

- 保持与现有前台深色工作台风格一致
- 商品页优先强调“可维护性”和“信息清晰”，不做后台式密集表格
- 新增按钮、编辑按钮、删除按钮均保持直观
- 删除操作必须带确认，避免误删

## 风险与约束

### 1. 商品列表可能混入公共商品

这是当前最主要的不确定点。第一版以前端过滤兜底，若联调不稳定，再补后端参数。

### 2. 后端字段较多，前端只接最小子集

本次必须避免一次性把全部商品字段做进 UI，否则范围会迅速膨胀。

### 3. 价格字段类型转换

前端表单输入天然是字符串，提交前必须做安全转换，否则容易与后端数值类型不一致。

### 4. 现有代码有部分历史中文乱码

本次新增页面与交互文案统一使用正常中文，尽量不在本轮扩散历史乱码问题。

## 结论

本次采用“前台主工作台内的商品管理视图”方案：

- 左侧栏新增 `商品`
- 主区域新增“我的商品”页面
- 列表展示当前用户拥有的商品
- 通过弹窗支持新增与编辑
- 删除需要确认
- 第一版仅支持最小字段：`名称 / 品牌 / 价格 / 主图链接 / 描述`

这个方案能最大化复用现有前台结构与后端 CRUD，范围可控，适合作为商品模块的第一期落地版本。
