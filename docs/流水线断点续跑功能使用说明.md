# 流水线断点续跑功能使用说明

## 🎯 功能概述
为视频和商品解析流水线添加了完整的持久化和断点续跑功能，用户可以查看解析进度、失败后手动重试，无需从头开始执行。

## 📋 新增功能

### 1. 解析状态管理
- 资产（视频/图片/音频）和商品都新增了解析状态字段：
  - `pending`: 待执行
  - `running`: 执行中
  - `completed`: 执行完成
  - `failed`: 执行失败
- 记录执行ID（`execution_id`）和错误信息（`parsing_error`）

### 2. 进度查询
- 支持实时查询解析进度，包括当前执行到第几个处理器、总进度百分比
- 可以通过资产ID/商品ID查询，也可以通过execution_id查询

### 3. 断点重试
- 解析失败后可以一键重试
- 自动从失败的处理器位置继续执行，跳过已经完成的步骤
- 避免重复计算，提高重试效率

## 🛠️ 代码使用示例

### 1. 同步执行并获取进度
```python
from backend.v1.app.rag.core.pipline import VideoParsingPipeline

# 创建流水线
pipeline = VideoParsingPipeline()

# 执行并持久化
result = pipeline.run_with_persistence({
    "video_id": 123,
    "video_url": "https://example.com/video.mp4",
    "object_name": "videos/test.mp4"
})

execution_id = result["execution_id"]

# 查询进度
status = VideoParsingPipeline.get_execution_status(execution_id)
print(f"进度: {status['current_processor_index'] + 1}/{status['total_processors']}")
print(f"状态: {status['status']}")

# 如果失败，重试
if not result["success"]:
    resume_result = pipeline.resume_execution(execution_id)
    print(f"重试结果: {resume_result['success']}")
```

### 2. 后台任务中使用
```python
# 在后台任务中执行解析，不阻塞主流程
def background_parse_task(asset_id):
    db = next(get_db())
    try:
        # 更新状态为运行中
        AssetDAO.update_asset(db, asset_id, {"parsing_status": "running"})
        
        # 执行解析
        result = AssetService._process_asset_parsing(db, asset_id, 2, "https://example.com/video.mp4")
        
        if result.get("error"):
            # 解析失败
            AssetDAO.update_asset(db, asset_id, {
                "parsing_status": "failed",
                "parsing_error": result["error"]
            })
        else:
            # 解析成功
            AssetDAO.update_asset(db, asset_id, {"parsing_status": "completed"})
    finally:
        db.close()
```

## 📊 数据模型变更

### 1. assets 表新增字段
| 字段名 | 类型 | 说明 |
|--------|------|------|
| parsing_status | varchar(20) | 解析状态：pending/running/completed/failed |
| execution_id | varchar(64) | 流水线执行ID，用于断点续跑 |
| parsing_error | text | 解析错误信息 |
| updated_at | datetime | 更新时间 |

### 2. products 表新增字段
| 字段名 | 类型 | 说明 |
|--------|------|------|
| parsing_status | varchar(20) | 解析状态：pending/running/completed/failed |
| execution_id | varchar(64) | 流水线执行ID，用于断点续跑 |
| parsing_error | text | 解析错误信息 |
| ai_features | json | AI解析结果特征 |

### 3. 新增 pipeline_executions 表
存储所有流水线执行记录和上下文快照，用于断点恢复。

## ⚡ 性能优化
- 持久化操作使用独立数据库会话，不影响业务逻辑事务
- 可配置持久化策略：
  - `enable_persistence=False`: 完全关闭持久化（原有行为）
  - `persist_after_each_processor=False`: 仅在错误时持久化，性能更好
- 上下文序列化使用JSON，存储效率高

## 🔍 故障排查

### 常见问题
1. **查询进度时没有progress_detail字段**
   - 说明execution_id无效或执行记录已被清理
   - 确认execution_id是否正确，或执行是否已超过7天（历史记录会自动清理）

2. **重试时提示"只有失败的解析任务可以重试"**
   - 只有状态为`failed`的任务可以重试
   - 如需重新执行成功的任务，使用`force=True`参数调用parse接口

3. **断点恢复失败**
   - 可能是流水线结构发生了变化（处理器数量不一致）
   - 系统会自动降级为重新执行完整流程

### 日志排查
- 流水线执行日志包含`[持久化]`前缀的日志
- 错误信息包含详细的堆栈跟踪，便于定位问题
